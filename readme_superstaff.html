🔧 คู่มือการใช้งานระบบ skinEPA V 2.40B สำหรับ Superstaff (ผู้ดูแลระบบ)

1. การจัดการฟิลด์ระบบ (R/W Field Management)

สิทธิ์ระดับรากฐานในการแก้ไขข้อมูลบุคลากรในชีต staff ผ่านหน้ากากแอปพลิเคชัน

  • Role & Status: ปรับเปลี่ยนระดับสิทธิ์ (เช่น resident3 ไป resident4) หรือเปลี่ยนเป็น ALUMNI เมื่อสิ้นสุดปีการศึกษา
  • การปลดล็อก LINE UID:
    หากผู้ใช้เปลี่ยนเครื่อง ให้ ลบข้อมูลช่อง LINE UID ให้ว่างเปล่า แล้วบันทึก ระบบจะตัดการเชื่อมโยงทันที เพื่อให้เริ่มล็อกอินด้วยสิทธิ์เริ่มต้นใหม่ได้
  • Advisor MDID:
    หากมีอาจารย์ที่ปรึกษาหลายท่าน ให้ใส่ MDID คั่นด้วยคอมมา เช่น 24105,11582 (ห้ามมีเว้นวรรค)

3. ระบบคัดแยกฐานข้อมูลรายปีการศึกษา (Multi-Year Data Isolation)

เพื่อรักษาประสิทธิภาพการทำงานและป้องกันปัญหาระบบหน่วง (Execution Timeout) ข้อมูลดิบจะถูกจัดเก็บแยกชีตตามปีการศึกษา (เช่น epa68, activity68) โดยมีโครงสร้างควบคุมดังนี้:

  • ฟังก์ชันสร้างชีตล่วงหน้าอัตโนมัติ (Time-driven Trigger): สคริปต์หลังบ้านถูกตั้งเวลาให้ทำงานโดยอัตโนมัติใน วันที่ 1 มิถุนายนของทุกปี (ล่วงหน้า 1 เดือนก่อนเปิดภาคเรียนวันที่ 1 กรกฎาคม) เพื่อตรวจคำนวณปีการศึกษาถัดไปและ Duplicate โครงสร้างตารางใหม่ออกมารอรับข้อมูลล่วงหน้า
  • การบำรุงรักษาชีตต้นแบบ (Core Templates Requirement): ห้ามทำการลบ ปรับเปลี่ยนชื่อ หรือทำลายโครงสร้างของตารางต้นแบบที่นำหน้าด้วยคำว่า template_ เป็นอันขาด เนื่องจากสคริปต์สลับปีจะใช้ตารางเหล่านี้เป็นแม่แบบขยายฐานข้อมูล ประกอบด้วย:
    template_epa, template_activity, template_eval360, template_activity_master, template_act_evaluation, และ template_epalogs
  • กลไก API Routing: ตัวแปรสัญญลักษณ์ academic_year (เลขปี พ.ศ. สองหลักท้าย เช่น 68, 69) จะถูกประทับลงใน Web Request Payload จากหน้า Frontend ทุกครั้ง เพื่อบังคับทิศทางให้ Google Apps Script เชื่อมโยงเซลล์ข้อมูลเข้าหาตารางประจำปีการศึกษานั้นอย่างแม่นยำ
